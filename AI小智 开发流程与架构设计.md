## 二、 架构与开发流程详解

### 2.1 应用层开发
应用层是整个智能体的核心逻辑，主要分为以下几个业务模块：

#### 2.1.1 OTA (在线升级)
* **初始化**
  * 初始化 `httpClient`
  * 注册 `httpClient` 的数据处理回调
    1. 解析收到的 JSON，将 websocket url、token、激活码解析出来
    2. 根据是否有激活码，给事件标志组发送激活成功/没有激活标志
    3. 如果数据错误，给事件标志组发送数据错误标志

#### 2.1.2 激活智能体
* **1. 设置请求头**
  * `device-id`：将 MAC 地址拼接成 `xx:xx:xx:xx:xx:xx` 格式
  * `client-id`：
    1. 打开 nvs
    2. 查看 `client_id` 是否在 nvs 中存在，存在则直接取出来使用
    3. 如果 nvs 中不存在，需要生成 uuid v4 作为 `client_id`
    4. 保存 `client_id` 到 nvs
    5. 提交 nvs
    6. 关闭 nvs
  * `content-type`
* **2. 发起请求**：根据接口需求，使用 cjson 拼成接口需要的 JSON 数据并设置到请求体发送
* **3. 状态等待**：等待标志位（Http请求错误、数据错误、激活成功、没有激活标志）
* **4. 结果处理**：如果是激活成功则退出循环；如果没有激活成功，后续做 LCD 显示告知用户需要根据指定激活码去激活
* **5. 异常重试**：没有激活、请求错误、数据错误需要等待 5s 后再次发请求查询（循环步骤 1-5）

#### 2.1.3 主体逻辑
1. 初始化智能配网
2. 注册 WIFI 连接成功回调（给公共事件标志组发送连接成功标志）
3. 初始化按键
4. 给 KEY1、KEY2 注册回调
5. 等待 WIFI 连接成功（通过事件标志组等待连接成功事件）
6. 创建需要的环形缓冲区（共 4 个）

#### 2.1.4 音频处理
1. 初始化 ES8311 并打开
2. 初始化 SR (语音识别模型)
3. 创建处理任务：
   * **任务A：采集数据**
     1. 读取 ES8311 麦克风数据
     2. 将读取到的数据放入环形缓冲区
   * **任务B：语音识别 (SR)**
     1. 获取每个通道需要处理的一帧数据大小【M】
     2. 获取通道个数【N】
     3. 计算出喂给 SR 语音模型的数据大小 = `M * N * sizeof(int16_t)`
     4. 从环形缓冲区取出指定大小的数据
     5. 将数据喂给 SR 模型
   * **任务C：处理识别结果**
     1. 取出 SR 的识别结果
     2. 判断是否唤醒：如果唤醒，将唤醒标志位置 1，同时调用唤醒回调
     3. 判断人声状态：如果已唤醒且人声状态变化，调用人声状态变化回调
     4. 语音缓存：如果唤醒并且处于说话状态，需要将语音存入缓冲

#### 2.1.5 音频编码
1. 获取需要传入的 PCM 音频数据大小，以及编码后音频大小
2. 根据第一步获取的两个大小，申请两块内存空间
3. 从环形缓冲区取出指定 PCM 大小的数据，存入上一步申请的内存中
4. 对音频进行编码
5. 将编码好的数据放入环形缓冲

#### 2.1.6 音频解码
1. 从环形缓冲取出 websocket 传来的数据
2. 循环将每帧音频数据进行解码
3. 将解码的数据通过 ES8311 播放
4. 释放环形缓冲内存

### 2.2 硬件接口层
* **按键**：初始化；按键1事件注册回调；按键2事件注册回调
* **ES8311 (音频芯片)**：初始化；读取麦克风数据；播放声音到扬声器
* **SR语音识别**：初始化；获取每个通道一帧数据大小；获取通道个数；喂数据；取结果
* **音频编码**：初始化；获取需传入的 PCM 大小及处理后的 opus 数据大小；执行音频编码
* **音频解码**：初始化；执行音频解码

---

### 2.3 驱动层
* **WIFI 驱动**：配网初始化；擦除配网信息；注册 WIFI 连接成功回调
* **HttpClient**：初始化；设置请求头；设置请求体；发起请求；注册接收数据处理回调
* **Websocket**：
  1. 初始化
  2. 添加请求体
  3. 发送文本/二进制数据
  4. 注册回调函数（处理接收的数据、处理 websocket 正常断开连接）
  5. 启动
  6. 停止
  7. 检查 websocket 是否连接状态
